## 0. 学ぶ前の状態

**状態①：エンジニアに言われた言葉**

「APIを繋げばいいだけだよ」

その時点で私はAPIが何かもわからなかった。繋ぐとは何か。どこに繋ぐのか。ぜんぶわからなかった。

**状態②：「このアプリ終わったかも」**

音楽療法アプリ「ねぃろ」開発中。診断フェーズのパラメータをLyria RealTimeに渡してようやく音が鳴った瞬間——

「なんか曲が切り刻まれてる」「スライサーがかかってる」「まじで終わったかも」

## 1. API通信の4要素

別のWebサービスを外から動かすには、4つのものが必要だとわかった。

| 要素 | 役割 | 例 |
|---|---|---|
| **APIエンドポイント** | アクセス先のURL（住所） | `https://api.example.com/generate` |
| **リクエスト形式** | 送るデータの構造（手紙の書き方） | `{"prompt": "calm jazz", "bpm": 80}` |
| **認証情報** | アクセス権の証明（会員カード） | Cookie / API Key |
| **実装手段** | 実際に送る方法 | Python / JavaScript |

```bash
# リクエストの基本形
POST https://api.example.com/generate
Authorization: Bearer YOUR_API_KEY
Content-Type: application/json

{"prompt": "calm jazz", "bpm": 80}
```

「繋げばいいだけ」の「繋ぐ」はこの4つを揃えることだった。

## 2. 公式APIのない世界で探す（Sunoの場合）

SunoAIには公式APIが存在しなかった。「それでも動かせないか」と試みた。

使った方法：**ブラウザの開発者ツール（Networkタブ）でリバースエンジニアリング**。

```
手順：
1. SunoAIをブラウザで使う
2. F12で開発者ツールを開く → Networkタブを選ぶ
3. SunoAIで音楽を生成する
4. Method = POST のリクエストを探す
5. Payloadタブでデータの中身を確認する
```

`v2-web/` というリクエストに、音楽生成に使われた全パラメータが入っていた。

```
エンドポイント: https://studio-api.prod.suno.com/api/generate/v2-web/
```

```json
// Payloadの中身（一部）
{
  "tags": "アコーディオンのメロディー, ダーティサウストラップ",
  "prompt": "[Verse 1]\n陽だまりの中で笑う君と...",
  "mv": "chirp-auk-turbo"
}
```

音響特性が個別パラメータではなく `tags` という文字列にまとまっていたのも発見だった。

## 3. 利用規約を先に読む

実装の前に、利用規約を確認した（AIに読んでもらった）。

```
Suno Terms of Service 禁止事項 #13：
engage in or use any data mining, robots, scraping,
or similar data gathering or extraction methods.
```

**スクレイピング・リバースエンジニアリングは明確に禁止。**

コードを1行も書く前に中止した。「まず利用規約を読む」がAPIを扱う最初のステップだと知った。

:::note warn
公式サポートのないAPIを使う場合、利用規約の確認は必須です。
:::

## 4. WebSocketとREST APIの違い

Lyria RealTime APIを使うことにした。ここで「APIには構造が2種類ある」ことを理解した。

| | REST API | WebSocket |
|---|---|---|
| 通信の仕組み | 1リクエスト → 1レスポンス | 接続を維持し続けてデータを流し続ける |
| 音楽生成の結果 | ファイルが1つできあがる | 音声データが細切れで届き続ける |
| インタラクティブ性 | 低い（生成後に変更不可） | 高い（パラメータを途中で変更できる） |
| 代表例 | Lyria 3 | Lyria RealTime |

ねぃろがLyria RealTimeを選んだ理由：「診断結果に合わせてリアルタイムで変化する音楽」を実現するため。一括生成より即時性を優先した。

```
REST API の流れ:
クライアント → リクエスト → サーバー → 音楽ファイル → 終了

WebSocket の流れ:
クライアント ←→ 接続を維持 ←→ サーバー
               音声データを流し続ける
```

## 5. 「動く」と「聴ける」は全然違う——スライサー問題

音が鳴った。でも——

```
「なんか切り刻まれてる」
「スライサーがかかってる」
「特にJ-Rockを選んだとき壊滅的」
```

WebSocketは音声データを細切れで届け続ける。その継ぎ目で音が切れていた可能性。ジャンル選択を追加してから壊れたから、ジャンルとボーカルハミングモードの相性問題かもしれない。エネルギーが高いと音の密度が上がるように設計していたから、それかもしれない。

原因がはっきりしなかった。

**試したこと：**

- ボーカルハミングモード → 通常モードに切り替え
- エネルギーと音の密度の連動 → いったん切り離し

全部やったらスライサーが消えた。

「何が一番効いたかは最後までわからなかった。でも直った」——そういう解決だった。

:::note info
「完璧に原因を特定してから直す」より「思い当たることを全部やる」方が早かった。
:::

## 6. プロンプト設計の失敗と2層設計

スライサーは消えた。でも次の問題：**J-Rockを選んでいるのに、なんか違う。**

原因がわかった。

```json
// ❌ 問題のあった設計
気分「悲しい」 → プロンプト: "チェロ、バイオリン"（楽器名を直接指定）
ジャンル「J-Rock」 → プロンプト: "ギター、ドラム"（楽器名を直接指定）

// → AIが「チェロとギターを同時に求められて」迷う
```

解決：ジャンルを選んだとき、楽器名の指定をやめて感情のキーワードだけ渡す。

```json
// ✅ Vibe × Genre の2層設計
Vibe（色・空気感）: "暗い" "メランコリック" "ゆっくりしたテンポ"
Genre（骨格）: "J-Rock"  // 楽器指定なし
```

「骨格（Genre）」と「色・空気感（Vibe）」を別々に設計した方が、意図した音楽が出やすい。今は18種類のVibeと21種類のGenreを用意している。

## 0. 学ぶ前の状態

医療系スタートアップで、出力画面の作成・修正を担当していた時期がありました。

テスト用ブランチで編集して、プルリクエストが承認されると本番環境が更新される開発フロー。

でも「テスト環境では正しく動いているのに、本番でなぜか不具合が出る」ことが続いた。どこを変えたかは正しかった。確認もした。なのに壊れた。理由が全くわからなかった。

代表から言われた言葉：**「Gitからやり直した方がいいですよ。」**

そのスタートアップは契約解除。当時はめっちゃ落ち込んだ。でも今は、あの言葉の意味がわかります。

## この記事でわかること

- Git と GitHub の違い
- コミット・プッシュ・プルリクエスト・マージの意味
- コンフリクトが起きる仕組みと、なぜ本番が壊れたのか

---

## 1. Git と GitHub は別物

まずここから混乱している人が多い。（私もそうでした）

| | Git | GitHub |
|---|---|---|
| 場所 | 自分のPC（ローカル） | ネット上（クラウド） |
| 役割 | 変更履歴を記録する | コードを保管・共有する |

「GitHub = Git の略称」ではなかった。Git は変更を記録する仕組みで、GitHub はその記録を保管する場所。

---

## 2. コミット・プッシュとは

```bash
git status                    # 現在の状態を確認（赤=未記録、緑=記録済み）
git add .                     # すべての変更をステージング
git commit -m "変更の説明"    # ローカルに記録する
git push origin main          # GitHubにアップロードする
```

```
コミット = 「ここまでの変更を記録した」というローカルへの保存
プッシュ  = 「GitHubに送って、ネット上でも保管する」というアップロード
```

コミットだけでは GitHub 上には何も反映されない。プッシュして初めてクラウドに出る。

---

## 3. ブランチとは「コードの並行世界」

本番を動かしたまま、新機能の開発や修正を別の流れで進められます。

```
main（本番・常に動いている）    ●―――――――――――●
                                │
                                └─●―●―●  feature/新機能（作業中）
```

チーム開発の基本的な流れ：

```
main（本番）
 └── develop（統合テスト環境）
       └── feature/自分の作業ブランチ（ここで開発する）
```

:::message alert
mainを直接編集するのは NG。本番が即変わってしまう。必ずブランチを切って作業する。
:::

---

## 4. プルリクエスト・マージとは

```
プルリクエスト（PR） = 「この変更を取り込んでください」というお願い
マージ               = ブランチの変更を別のブランチに統合すること
```

チームでのフロー：

1. feature ブランチで開発する
2. プルリクエストを送る
3. レビュアーが確認・承認
4. マージ（develop または main に統合）

---

## 5. コンフリクトとは、なぜ起きるのか

コンフリクトとは「同じファイルの同じ箇所を、異なる内容で変更したとき、どちらを採用すべきかわからなくなる状態」です。

```
自分：「このファイルのこの行を A に変えた」
他の人：「そのファイルの同じ行を B に変えた」
Git：「どっちが正しいの？わからないのでエラー（コンフリクト）にします」
```

---

## 6. 私の失敗の正体

当時の自分が何をしていたか、今ならわかります。

```
main（1週間前の古い状態）  ← ここからブランチを切っていた
 └── 自分のブランチ（古いコードを元に作業）

develop（最新・他の人の変更が入っている状態）
```

**ブランチのSourceをmainにしていた。**

本来は develop の最新から切るべきところ、古い main から切っていた。その間に他の人の変更が develop に入り、自分のコードと衝突していた。

テスト環境でOKだったのは「自分のブランチ内では整合性が取れていたから」。
本番で壊れたのは「develop にマージした瞬間にコンフリクトが発生したから」。

「正しくやったつもり、テストでも通った、でも本番が壊れた」——全部説明がつく。

---

## まとめ

| 学んだこと | ポイント |
|---|---|
| Git と GitHub の違い | Git = ローカルの記録、GitHub = クラウドの保管場所 |
| コミットとプッシュ | commit = ローカルに記録、push = GitHubにアップロード |
| ブランチの役割 | 本番を動かしながら開発できる並行世界 |
| プルリクエスト | 「この変更を取り込んで」というお願い |
| コンフリクト | 同じ箇所の違う変更による衝突。古いコードから始めると起きやすい |

「Gitからやり直した方がいい」の意味が、今はわかります。

コンフリクトとは何か。最新のコードから始めることがなぜ重要か。この2つを知らなければ、正しくやったつもりでも本番は壊れる。

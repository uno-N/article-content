# 「このアプリ、終わったかも」——AI音楽生成で壊滅した日の話

みなさんこんにちは。平凡な訪問看護師が、アプリ開発の実装過程を掲載していく、「アプリ開発日記Vol.7」です。失敗や成功を繰り返していく過程をお楽しみください。

前回は診断フェーズのUI設計・セキュリティについて書きました。

今回は、診断結果や音響パラメータを実際にLyria RealTime APIに渡して音楽を生成するお話です。でも実際に鳴らしてみたら——

「なんか曲が切り刻まれてる」「まじでこのアプリ終わったかも」、、、

パラメータを増やした途端に起きたスライサー問題。やっと辿り着いたLyria RealTime APIだったのに、、原因と解決のお話を書いていきます。

---

## 0. はじめに：Lyria RealTimeとLyria 3の違い

ねぃろで使っているのは **Lyria RealTime API** です。みなさんが知っている「Lyria」は、GeminiやAI Studioに搭載されている **Lyria 3** のことが多いと思うので、ここで整理しておきます。

![Lyria RealTimeを使用しているGeminiの音楽生成](https://i.gyazo.com/4c3c8b984ab3d24c9ddb4335341329d6.png)

ねぃろでは、診断結果のパラメータ（BPM・スケール・気分プロンプト）をリアルタイムで渡しながら音楽を生成するため、インタラクティブなRealTimeを選んでいます。

Lyria 3は円滑にボーカル生成が行えるため、一聴ではクオリティが高く感じます。ただ、ねぃろは「今の状態に合わせてリアルタイムで変化する音楽」がコンセプトなので、一括生成より即時性を優先しています。

| | **Lyria 3** | **Lyria RealTime** |
|---|---|---|
| 用途 | テキストプロンプトから高品質な音楽を一括生成 | リアルタイムにストリーミングで音楽を生成 |
| API形式 | REST（1リクエスト → 1音楽ファイル） | WebSocket（音声データが細切れで届き続ける） |
| インタラクティブ性 | 低い（生成後に変更不可） | 高い（パラメータを途中で変更できる） |
| 主な用途 | 楽曲制作、BGM一括生成 | インタラクティブな音楽体験、ゲーム・アプリへの組み込み |
| 品質 vs 即時性 | 品質優先 | 即時性優先 |

## 1. 診断フェーズで何を測り、どう渡すか

### 1-1. ユーザーが選ぶこと

診断フェーズでは、次の4つを測ります。

1. **今の気分**（気分の円環を元にした気分選択で音響反映）
2. **エネルギー**（創造を促すべきか、転化に向かうべきか）
3. **タップテンポ**（BPMを手で叩いて入力、リズム複雑度で心情を反映）
4. **内向性・外向性スコア**（音楽傾向を反映）

### 1-2. その値がLyriaに渡される

集めた値は、Lyria RealTime APIのパラメータに変換されます。

| 診断値 | Lyriaへの変換 |
|--------|----------------|
| タップテンポ | `bpm`（数値で直接指定） |
| タップの揺らぎ（CV値） | リズム複雑度テキスト（安定〜シンコペーション） |
| 気分カテゴリ（ヘヴナー） | 音階（スケール）＋ムードテキストプロンプト |
| エネルギー | `brightness`（高音域の明るさ）の基準値 |
| E/Iスコア | 3パターン目のジャンル自動選択＋音の明るさ調整 |

診断のあと、ユーザーが **Vibe**（lo-fi・cinematic・dreamy などつまりバイブス）と **Genre**（Rock・Jazz・Ambient などつまりジャンル）を選択します。

### 1-3. 音楽が生成される

今回は3パターンの音楽が生成され、その中から一番好きなものを選択できる形式にしました。

①診断結果のみ反映
②VibeやGenreも反映
③E/Iスコアから自動算出したGenreを反映

それぞれ異なるコンセプトで。今の状態そのものが反映されるようにしました。

でも、実際に音が鳴った瞬間——

## 2. このアプリおじゃんになるかも

……ディレイがかかってる、、？というかスライサーがかかってる。

音がぶつぶつ途切れる。切り刻まれてる感じ。特にJ-Rockを選んだときが壊滅的だった。

単純に音響パラメータ反映させた時は問題なかったのになぜこんなことになったんだ？？

---

### 2-1. 試しまくった

何が原因かはっきりしなかった。

Lyria RealTime APIは音をリアルタイムのかたまりで届ける仕組みで、そのかたまりの継ぎ目で音が切れている可能性。ジャンル選択を追加してから壊れたから、ジャンルとボーカルハミングモードの相性問題かもしれない。エネルギーが高い人ほど音の密度を上げていたから、それかもしれない。

思い当たること全部試した。

- ボーカルハミングモード → 通常モードに切り替えた
- エネルギーと音の密度の連動 → いったん切り離した

どれが一番効いたかは正直わからない。全部やったら、スライサーが出なくなった。

「全部つぶしたら直った」という解決の仕方だった、、笑。

---

### 2-2. まだなんか違う

スライサーは消えた。でも次の問題が出た。

J-Rockを選んでいるのに、なんか、違う。

調べてみると、気分カテゴリのプロンプトに「チェロ」「バイオリン」という楽器名が入っていた。「悲しい気分」を表現するために自分で書いたものだ。

チェロとギターを同時に求めているから、AIも迷う。そりゃそうか、という話だった、、。

ジャンルを選んだときは楽器の指定をやめて、感情のキーワードだけ渡すことにした。「暗い」「メランコリック」「ゆっくりしたテンポ」みたいな形で。

**これが「Vibe × Genre の2層設計」につながった。**

「骨格（Genre）」と「色・空気感（Vibe）」を別々に指定する方が、意図した音楽が出やすい。今は18種類のVibeと21種類のGenreを用意してある。

---

## 3. 「今の状態」から生まれる3パターンの音楽

ねぃろでは3パターンの音楽を生成する。

- **パターンA**：診断結果のみ（ジャンルなし、純粋な音響出力）
- **パターンB**：ユーザーが選んだVibe + Genre
- **パターンC**：E/Iスコアから自動で選ばれたジャンル

3つとも「今のあなたの状態」から出てくるように設計した。

パターンCに使ったのが、診断フェーズを設計するときから根拠にしていたRentfrow & Gosling（2003）の研究。音楽心理学の研究で、「どんな性格の人がどんなジャンルを好む傾向があるか」を統計的に分析したもの。

- 内向的な人 → Jazz、Classical、Folk、Ambient系
- 外向的な人 → Pop、Electronic、Hip-hop、R&B系

これをそのままコードに落とした。「なんとなくJazzかな」じゃなくて、論文の裏付けがある。

「あなたに合う音楽」を根拠つきで出せるアプリになった（ちょっと盛りました）。

---

## 4. 学んだこと

「動く」と「聴ける」は全然違う。

APIが返してくる音声データは正しくても、パラメータ1つで全然違う音になる。公式ドキュメントに設定の仕方は書いてあっても、複合的に触ったバランス設定値の「正解」がどこにも書いていないから、試すしかない。

原因がわからなくても、試せることは全部試す。

何が効いたか最後までわからなかったけど、それでいいと思った。完璧に原因を特定してから直すより、思い当たることを全部やる方が早かった。

もう1つ気づいたのは、根拠を持たせることの気持ちよさ。

sunoAI、MurekaAI、Udioではできないことが、Lyriaではできる。論文を読んでコードに落とすのは時間がかかるけど、「なぜこの音楽が出てくるのか」を説明できるアプリになった。ただランダムに出てくるより、ずっと意味がある。

創造にクオリティは大切だが、私は芸術を見るとき、空気感とコンテキストの方を大事にしてみる。だから、Neiroも同じだと思ってもらいたい。（単に音楽だけのクオリティはさほど高くないと言いたくないだけ）

触ってもらったら「めっちゃ良くなった」と言ってもらえて、それが一番嬉しかった。

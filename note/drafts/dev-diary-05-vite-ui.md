## 0. まず最初にViteに乗り換えた

### 0-1. またCRAの話

#2の記事の末尾にこっそり追記していたんですが、Create React App（CRA）が2025年2月に非推奨になりました。

ねぃろの最初の環境構築はCRAで行っていたのですが、今回新しくフロントエンドを作り直すタイミングで、ちゃんとViteに乗り換えることにしました。

CRAはコマンドを実行してから「終わったかな？」と思ってもまだ終わってなくて、体感3〜5分は待たされます。Viteはほぼ一瞬。この差は体験として全然違う。

```bash
# 旧：Create React App（非推奨）
npx create-react-app my-app

# 新：Vite（こっちを使おう）
npm create vite@latest my-app -- --template react
cd my-app
npm install
npm run dev
```

特に大きな問題もなく乗り換えられました。

## 1. 診断フェーズの実装に取り掛かる

### 1-1. MVP仕様書を渡したのに……

前回作ったMVP仕様書があります。気分・エネルギー・リズム・性格の4ステップを診断してもらう、アプリの根幹の部分です。

Claude Codeに「診断フェーズを実装して」とお願いしたところ……なんか想像と全然違うものが出てきた。

仕様書は手元にあるのに、それを読まずに「なんとなくこんな感じかな」で実装されてしまっていた。

**そりゃそうだ。パス指定してなかった。**

Claude Codeはファイルを自動で読みに行くわけではなく、「このファイルを参照して」と明示的に伝えないといけません。

```
src/MVP_spec.md を読んで、診断フェーズを実装して
```

パスを指定して再度依頼すると、ちゃんとMVP仕様書の内容を読み込んで、意図通りのコンポーネント構成を出してくれた。

### 1-2. 構成はOKだった

出てきた構成はこんな感じ。

```
src/
├── components/
│   ├── Welcome.jsx           # ウェルカム画面
│   ├── DiagnosisFlow.jsx     # 診断フロー管理
│   └── steps/
│       ├── MoodSelect.jsx    # Step1: 気分選択
│       ├── EnergySlider.jsx  # Step2: エネルギー値
│       ├── TapTempo.jsx      # Step3: テンポ入力
│       └── PersonalityCheck.jsx # Step4: 性格タイプ
```

4ステップが独立したコンポーネントになっていて、`DiagnosisFlow.jsx` が状態管理しながらステップを切り替える構造。これは想定通りでOKでした。

## 2. UIに取り掛かる、が迷子になる

### 2-1. デザインどうしよう問題

コンポーネントの骨格ができたら、次はUI。

看護師が作ってる音楽療法アプリのUIってどんなんがいいんだろう。「癒し系にしたい」という想いはあるけど、それをコードにどう伝えるか。毎回Claude Codeに「いい感じにして」と伝えると出てくるのはのっぺりした白いUIで、なんか違う。

そんな時にXでこんなツイートを見かけました。

> 「DESIGN.mdって知ってますか？有名プロダクト（Notion、Linear、Figmaなど）のデザインシステムをまとめたMarkdownファイルを集めたリポジトリがあって、AIに渡すと一発でそのスタイルで実装してくれる」

**え、それやりたい。**

### 2-2. DESIGN.mdって何？

awesome-design-md というGitHubリポジトリに、有名プロダクトのデザイン仕様をMarkdownで書いたファイルが集められています。

NotionのUI、LinearのUI……全部Markdownで「色は何、フォントは何、ボーダーはこう」と書いてある。それをそのままAIに渡すと、そのスタイルで実装してくれる。

なるほど。AIに「いい感じに」と伝えるんじゃなくて、「何がいい感じなのか」を先に定義しておけばいい。

### 2-3. 自分用のNEIRO-DESIGN.mdを作ってもらった

じゃあねぃろのデザイン仕様書を作ろう、ということでClaude Codeと一緒に `NEIRO-DESIGN.md` を作りました。

「アイスブルーのグラデーション、ガラスカード、浮遊するオーブ」というビジョンを伝えて、最初の版を生成してもらった。

## 3. 3回スタイルを試した

### 3-1. まずアイスブルーで試す

最初のビジョン通り、アイスブルーのグラデーション背景にガラスカードが浮かぶUI。

悪くない。でも「もう少し整理されてる感じがほしいな」と思って、有名プロダクトのスタイルで試してみることにした。

### 3-2. Notionスタイルを試す

DESIGN.mdを渡してNotionスタイルで実装してもらった。

温かみのあるオフホワイト、カードに柔らかい影、読みやすい。
「これもいいね」となったけど、ガラスっぽさが消えてしまった。

### 3-3. Linearスタイルを試す

次にLinear。線が細く、ウェイト510/590の繊細なタイポグラフィ、輝度ステッキング。

「アイスブルーも良かったんだけど、Linearもいいなってなっちゃったんですよね」

---で、「じゃあ両方合体させたらどうなる？」となった。

### 3-4. B-style（アイスブルー × Linear）で確定

アイスブルーのグラデーション背景 × Linearのウェイト体系・ボーダー設計。

これが一番しっくりきた。「癒し」と「信頼感」が両立した感じ。

```css
/* 背景 */
background: linear-gradient(135deg, #1a3a5c 0%, #2d6a9f 50%, #4a90c4 100%);

/* ガラスカード */
background: rgba(255, 255, 255, 0.08);
backdrop-filter: blur(20px);
border: 1px solid rgba(255, 255, 255, 0.12);
```

この仕様をNEIRO-DESIGN.mdに書き込んで確定。次回から同じスタイルで書けるようにした。

## 4. そうこうしているうちに、ツイートがバズった

UIを試行錯誤していた頃、Xに投稿していた開発日記のつぶやきが予想外に伸びた。

看護師が音楽療法アプリを作っているという話、Eysenckの内向外向と脳の基底覚醒レベルの話（落ち込んでる人に明るい曲は逆効果、みんな知ってるようで神経科学的に説明するとこうなる、という話）。

思ったより多くの人が「知らなかった」「面白い」と言ってくれた。

**看護師×個人開発×音楽療法という組み合わせで、誰かの役に立てる情報を発信できるかもしれない。**

そう思ったら、開発の方向性も少しクリアになった気がした。

## 5. まとめ

### 5-1. 今回の学び

- ✅ CRAは非推奨 → Viteで起動が圧倒的に速い
- ✅ Claude Codeにはパスを明示的に渡す → 「この仕様書を読んで実装して」
- ✅ DESIGN.mdを作ると「いい感じに」が具体的になる
- ✅ スタイルは試して比べるのが一番早い

### 5-2. つまずきポイントと解決法

**MVP仕様書が反映されない：**
Claude Codeにファイルを読んでほしい時は、ファイルパスを明示して渡す。「なんとなく実装して」ではなく「この仕様書を読んで実装して」。

**「いい感じにして」が伝わらない：**
DESIGN.mdを作って仕様を定義する。色・フォント・ボーダー・コンポーネントのCSSを先に決めておく。

### 5-3. 次回の挑戦は、、？

音楽生成フェーズへの接続。
診断フェーズで収集した気分・エネルギー・テンポ・性格タイプを、Lyria RealTime APIのパラメータに変換して、実際に音楽を生成する。

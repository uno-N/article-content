### 0-2. 「DevOps」って何。

「DevOps」というのは、開発（Development）と運用（Operations）を一体で考える思想で、この診断が測っているのはまさにそれ。

- コードを書く（開発）
- 壊れないか確認する（品質保証）
- 本番で動かし続ける（運用）
- それを自動化・効率化する（自動化基盤）

この全体をDevOpsと呼びます。
開発者たるもの、めっちゃ大事そうじゃないですか。

その結果は、

> 総合: C ランク（31%）
> カテゴリ別では自動化基盤とインフラ管理が特に低く（どちらも17%）、AI連携環境と品質保証はやや高め（44%・42%）。

「やばいかも、、、」
結果を見て、個人開発にも適用できるものを実行してみようと思った。

## 1. やったこと

[![Image from Gyazo](https://i.gyazo.com/c0a8e4141eb84bd0636ae5e1e3dbc5ea.png)](https://gyazo.com/c0a8e4141eb84bd0636ae5e1e3dbc5ea)

### 1-1. `.gitignore` と `.env` を整えた

APIキーを安全に扱うための基本設定です。`.gitignore` に `.env` が入っていなければGitHubに上がってしまいます。一行追加するだけで防げます。

Viteを使っている場合、環境変数には `VITE_` という接頭辞が必要という罠もありました。これがないとReactのコードから読めません。

### 1-2. `CLAUDE.md` にセキュリティルールを書いた

Claude Codeを使っていると、APIキーの話をした途端に `.env` を勝手に読みに行く、という事態が発生しました。会話データにキーが表示されてしまって焦りました。

対策は「読むな」と書くだけ。CLAUDE.md に一行追加したら解決しました。

### 1-3. `DEVOPS.md` を作って用語を整理した

DevOpsの用語って、聞いたことはあっても自分のプロジェクトに当てはめると何をすればいいかわからないことが多いです。「ねぃろの場合はどうするか」を一覧にして保存しました。

### 1-4. Sentry（エラー監視ツール）を導入した

アプリでエラーが起きたとき、ユーザーより先に自分にメールが来る仕組みです。無料で使えます（月5,000件まで）。

インストールしたら脆弱性の警告が出て焦りました。が、詳細を読んだら「開発サーバー限定」の問題でした。本番には関係ない。一コマンドで修正完了しました。

でも放置はNG。

## 3. まとめ

### 3-1. 今回の学び

- ✅ `.gitignore` に `.env` を1行追加するだけでAPIキー漏洩を防げる
- ✅ Viteでは環境変数に `VITE_` プレフィックスがないとReactから読めない
- ✅ Claude Codeには「.envを読むな」と書くだけで誤アクセスを防止できる
- ✅ Sentryの脆弱性警告は「開発環境限定」でも放置NG。1コマンドで直せるなら直す
- ✅ DevOps成熟度診断はCランクだった。全部は無理でも、できることから1つずつ

### 3-2. コマンド・コード一覧

**Sentry導入**
```bash
npm install @sentry/react @sentry/vite-plugin
```

**脆弱性の確認・修正**
```bash
npm audit        # 脆弱性の確認
npm audit fix    # 自動修正（--force は原則使わない）
```

**`.gitignore` に追加する行**
```
.env
.env.local
.env.*.local
```

**Viteの環境変数（接頭辞 `VITE_` が必須）**
```
VITE_GEMINI_API_KEY=your_api_key_here
```

**CLAUDE.md に追加するセキュリティルール例**
```
## セキュリティ
- .envファイルを読まないこと
- APIキーはコードに直接書かないこと
```
